Regras de tamanho dos sprites

Conforme o comentário na linha 23 de sprites.ts:

▎ // Each sprite is 5 lines tall, 12 wide (after {E}→1char substitution).

Dimensões fixas:

- Largura: 12 caracteres (depois que {E} é substituído pelo olho, que é 1  
  char)
- Altura: 5 linhas por frame

Slot do chapéu (linha 0):

- A linha 0 é reservada para o chapéu — deve estar vazia nos frames 0 e 1
- O frame 2 (o "fidget") pode usar a linha 0 para detalhas como fumaça,  
  antena, etc. (ex: octopus frame 2 tem o )
- Se não houver chapéu e todos os frames tiverem a linha 0 vazia, a linha é  
  removida (linhas 497-498), reduzindo a altura de 5 para 4

Hats também seguem 12 chars de largura (ex: crown = \^^^/ , tophat =  
 [___] ).

### Lógica de renderização

O buddy tem **dois modos**:

1. **Modo narrow** (terminal < 120 colunas + fullscreen ativo): mostra só a face (`renderFace`) em 1 linha
2. **Modo full** (terminal >= 120 colunas OU fullscreen desativado): mostra o sprite 24x10 completo

### Problema encontrado

Com `isFullscreenEnvEnabled() = false` (padrão), **o sprite 24x10 renderiza mesmo em terminais de 80 colunas**. Isso porque:

- `companionNarrow = true` (80 < 120)
- Mas `!(companionNarrow && isFullscreenEnvEnabled())` = `!(true && false)` = `true`
- Resultado: sprite completo de 24 chars de largura num terminal de 80 cols

O sprite original de 12x5 cabia bem, mas o de 24x10 pode causar problemas de layout.

`★ Insight ─────────────────────────────────────`

- O placeholder `{E}` tem 3 caracteres no código-fonte mas vira 1 char após substituição — por isso as linhas com `{E}` medem 28 chars antes da substituição mas 24 depois
- O `MIN_COLS_FOR_FULL_SPRITE = 120` foi calibrado para o sprite de 24 chars + nome + padding + bolha de fala
- Em modo non-fullscreen, a lógica ignora o narrow mode e renderiza o sprite completo, o que pode quebrar o layout em terminais estreitos
